# 〇❌パーティゲーム Webアプリ 詳細仕様書

---

## 1. プロジェクト概要

- **アプリ名**: 〇❌パーティゲーム
- **目的**: 大学パーティ用の〇❌ベットゲームを招待制・認証付きWebアプリとして提供
- **対象ユーザー**: 招待コードを持つ生徒チームメンバー + 管理者

---

## 2. 技術スタック

| レイヤー | 技術 |
|----------|------|
| フロントエンド | Next.js 14 (App Router) |
| スタイル | Tailwind CSS + カスタムCSS |
| 認証 | Supabase Auth（Google OAuth） |
| DB | Supabase（PostgreSQL） |
| ホスティング | Vercel |
| 追加ライブラリ | なし（Supabase SDK のみ） |

**Supabase Auth を使うことで NextAuth は不要。**  
Supabase の Google OAuth + Row Level Security (RLS) で認証・認可を完結させる。

## 3. 認証・招待コードシステム

### 3-1. 招待コードの仕様

- 管理者のみが招待コードを生成・一覧表示できる
- コードはランダム英数字8文字（例: `X7K2PQ9R`）、ハイフンなし
- 有効期限: 生成から7日間（管理者が任意変更可能）
- 使い捨て: 1コード1アカウントのみ有効
- 使用済みコードは `used_by` に user_id が入り無効化される

### 3-2. ログインフロー

```
① ユーザーがアプリにアクセス
   → 未ログインなら「招待コード入力画面」を表示

② 招待コードを入力して「確認」ボタン
   → Supabase DB の invite_codes テーブルを検索
   → 無効 or 使用済み or 期限切れ → エラーメッセージ表示・弾く
   → 有効 → コードをセッションストレージに一時保存し Google ログインへ進む

③ Google OAuth（Supabase Auth）でログイン
   → コールバック時にセッションストレージのコードを再検証
   → 有効なら invite_codes.used_by に user_id をセット・used_at を記録
   → users テーブルにユーザー情報を Upsert
   → マイページへリダイレクト

④ 2回目以降のログイン
   → Supabase セッションが有効なら招待コード入力をスキップ
   → 直接マイページへ
```

### 3-3. 管理者設定

- `users.is_admin = true` のユーザーのみ管理者パネルにアクセス可能
- 初期管理者は Supabase ダッシュボードまたはシードSQLで手動設定

## 4. DBスキーマ（Supabase PostgreSQL）

### `users` テーブル

```sql
CREATE TABLE users (
  id            UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email         TEXT UNIQUE NOT NULL,
  display_name  TEXT,
  avatar_url    TEXT,
  is_admin      BOOLEAN DEFAULT FALSE,
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  last_login_at TIMESTAMPTZ DEFAULT NOW()
);
```

### `invite_codes` テーブル

```sql
CREATE TABLE invite_codes (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code        TEXT UNIQUE NOT NULL,
  created_by  UUID REFERENCES users(id),
  used_by     UUID REFERENCES users(id) DEFAULT NULL,
  used_at     TIMESTAMPTZ DEFAULT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

### `game_sessions` テーブル

```sql
CREATE TABLE game_sessions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
  chip_base       NUMERIC NOT NULL DEFAULT 1,  -- ユーザー設定の基準値（1チップ＝何円/ドル）
  started_at      TIMESTAMPTZ DEFAULT NOW(),
  ended_at        TIMESTAMPTZ DEFAULT NULL,
  total_profit    NUMERIC DEFAULT 0,           -- セッション終了時の累計損益（chip単位）
  o_count         INTEGER DEFAULT 0,           -- セッション内の〇総数
  x_count         INTEGER DEFAULT 0            -- セッション内の❌総数
);
```

### `sets` テーブル

```sql
CREATE TABLE sets (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id          UUID REFERENCES game_sessions(id) ON DELETE CASCADE,
  set_index           INTEGER NOT NULL,         -- セッション内の連番（1始まり）
  results             TEXT NOT NULL,            -- 例: "OOXOXXO"（7文字固定、O=〇 X=❌）
  wins                INTEGER NOT NULL,         -- 〇の数
  losses              INTEGER NOT NULL,         -- ❌の数
  overshoot           INTEGER NOT NULL,         -- 負け越しマスの値（0以上）
  slashed             BOOLEAN DEFAULT FALSE,    -- 斜線フラグ
  used_unit_idx       INTEGER NOT NULL,         -- このセットで実際に使ったSEQインデックス
  next_unit_idx       INTEGER NOT NULL,         -- 次セット用SEQインデックス（チップ単位行の表示値）
  set_profit          NUMERIC NOT NULL,         -- このセットの損益（chip単位）
  cumulative_profit   NUMERIC NOT NULL,         -- このセットまでの累計損益（chip単位）
  created_at          TIMESTAMPTZ DEFAULT NOW()
);
```

### RLS ポリシー

```sql
-- users: 本人のみ読み書き / 管理者は全件読み取り可
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
CREATE POLICY "self_read" ON users FOR SELECT USING (auth.uid() = id);
CREATE POLICY "admin_read" ON users FOR SELECT USING (
  EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = TRUE)
);

-- game_sessions: 本人のみ / 管理者は全件読み取り可
ALTER TABLE game_sessions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "self_session" ON game_sessions FOR ALL USING (auth.uid() = user_id);
CREATE POLICY "admin_session" ON game_sessions FOR SELECT USING (
  EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = TRUE)
);

-- sets: 本人のみ / 管理者は全件読み取り可
ALTER TABLE sets ENABLE ROW LEVEL SECURITY;
CREATE POLICY "self_sets" ON sets FOR ALL USING (
  EXISTS (SELECT 1 FROM game_sessions WHERE id = session_id AND user_id = auth.uid())
);
CREATE POLICY "admin_sets" ON sets FOR SELECT USING (
  EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = TRUE)
);

-- invite_codes: 管理者のみ生成・閲覧
ALTER TABLE invite_codes ENABLE ROW LEVEL SECURITY;
CREATE POLICY "admin_invite" ON invite_codes FOR ALL USING (
  EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = TRUE)
);
```

## 5. コアゲームロジック（正確に実装すること）

### 5-1. チップ単位数列（SEQ）

インデックス0から順に以下を使用（変更不可）:

```javascript
const SEQ = [
  1, 2, 3, 5, 7, 9, 11, 13,
  16, 19, 22, 25, 28, 31,
  35, 39, 43, 47, 51, 55,
  60, 65, 70, 75, 80, 85, 90, 95, 100,
  106, 112, 118, 124, 130, 136, 142, 148, 154, 160,
  170, 180, 190, 200, 210, 220, 230, 240, 250
];
```

### 5-2. 1セットの定義

- 7ターン連続の〇❌結果を1セットとする
- 常に〇にbetする（❌にはbetしない）
- 7回すべて入力完了した時点でセット確定・DB保存

### 5-3. 勝敗判定

```javascript
wins   = results.filter(r => r === 'O').length  // 0〜7
losses = 7 - wins
diff   = wins - losses  // 必ず奇数になる（7回なので引き分けなし）

// diff > 0 → 勝ち（4勝3負以上）
// diff < 0 → 負け（3勝4負以下）
```

### 5-4. 負け越しマス（overshoot）計算

```javascript
// 前セットの overshoot を prevOvershoot とする（初回は 0）
newOvershoot = prevOvershoot - diff
if (newOvershoot < 0) newOvershoot = 0
// ※ overshoot は 0 以下にはならない
```

### 5-5. 斜線（slashed）処理

勝ち（diff > 0）が確定した直後に実行:

```javascript
for (let i = 0; i < sets.length; i++) {
  if (!sets[i].slashed && sets[i].overshoot > newOvershoot) {
    sets[i].slashed = true  // DB も UPDATE する
  }
}
```

- 斜線が引かれたセルは UI 上でピンクの横線を表示する
- 斜線セルの「負け越し」と「チップ単位」の数字はグレーアウトし横線を引く

### 5-6. 次セットのbet単位（next_unit_idx）決定

```javascript
if (diff < 0) {
  // 負け → 1段階UP
  nextUnitIdx = Math.min(usedUnitIdx + 1, SEQ.length - 1)

} else if (diff > 0) {
  // 勝ち → 現セッション内の非斜線セットから overshoot が一致するものを後ろから検索
  let found = -1
  for (let i = sets.length - 1; i >= 0; i--) {
    if (!sets[i].slashed && sets[i].overshoot === newOvershoot) {
      found = sets[i].next_unit_idx
      break
    }
  }
  if (found >= 0) {
    nextUnitIdx = found
  } else {
    // 一致なし → 非斜線セットの中で overshoot が newOvershoot より大きく最も近いものを使用
    let best = -1, bestDiff = Infinity
    for (let i = 0; i < sets.length; i++) {
      if (!sets[i].slashed && sets[i].overshoot > newOvershoot) {
        const d = sets[i].overshoot - newOvershoot
        if (d < bestDiff) { bestDiff = d; best = sets[i].next_unit_idx }
      }
    }
    nextUnitIdx = best >= 0 ? best : 0
  }
}
```

### 5-7. 損益計算

```javascript
// used_unit_idx = このセットが始まる前の currentUnitIdx（前セットの next_unit_idx）
// 初回セットの used_unit_idx = 0（SEQ[0] = 1チップ）

setProfit        = diff * SEQ[usedUnitIdx]           // このセット単体の損益（chip単位）
cumulativeProfit = prevCumulativeProfit + setProfit   // 累計損益（chip単位）

// 実際の金額表示 = chip単位の値 × chip_base（ユーザー設定の基準値）
```

### 5-8. チップ単位行の意味（重要）

- チップ単位行に表示する値 = **次のセットで使うbet単位** = `SEQ[next_unit_idx]`
- 損益計算に使う単位 = **このセットで実際に使った単位** = `SEQ[used_unit_idx]`
- `used_unit_idx` = 前セットの `next_unit_idx`（初回は 0）

## 7. ゲームセッション仕様

### 7-1. セッションの開始・終了

- マイページに「**ゲーム開始**」ボタン
- 押すと `game_sessions` に新規レコード作成・`started_at` を記録
- セッション中は「**ゲーム終了**」ボタンを表示
- 終了ボタンを押すと `ended_at` を記録・`total_profit` を更新
- 終了後はそのセッションの結果サマリーを表示

### 7-2. 〇❌パーセント表示

セッション開始から現時点までの全ターン（7×セット数）の集計:

```
〇: XX%（XX回）
❌: XX%（XX回）
```

### 7-3. セッション中の表示

- **入力パネル**（画面上部固定）:
  - 〇ボタン・❌ボタン・戻すボタン
  - 入力中のターン（1/7〜7/7）を7つのスロットでリアルタイム表示
  - 〇/❌を押した瞬間に表のその列に即時反映
- **bet情報バー**:
  - 「今セットのbetチップ単位: X チップ（= Y 円）」
  - 「累計損益: ±Z チップ（= ±W 円）」
- **管理表**（ノート形式）:
  - シート1: 第1〜20セット
  - シート2: 第21〜40セット
  - シート3: 第41〜60セット
  - 各シートは横20列・縦12行（7ターン行 + 勝敗 + 負け越し + チップ単位 + 純利益）

## 14. セキュリティ要件

- 招待コードの検証は必ずサーバーサイド（Server Action）で行う
- `SUPABASE_SERVICE_ROLE_KEY` はサーバーサイドのみで使用し、クライアントに露出させない
- RLS により、他ユーザーのデータへのアクセスをDB層でブロック
- 管理者チェックは RLS ポリシーとサーバーサイド両方で実施

## 15. Claude Code への実装指示

1. `supabase/migrations/001_init.sql` に全テーブル + RLS ポリシーのマイグレーションSQLを作成すること
2. `lib/gameLogic.ts` にSEQ配列と全計算関数を集約し、フロントとサーバーで共有すること
3. ゲームロジックの計算順序は必ず以下の通りとすること:
   1. セット結果確定
   2. overshoot 計算
   3. 斜線処理（該当セットの slashed を true に更新）
   4. next_unit_idx 決定
   5. 損益計算（set_profit・cumulative_profit）
   6. DB保存（sets テーブルへ INSERT、斜線対象セットを一括 UPDATE）
4. DB保存はセット確定ごとにリアルタイムで行い、ページリロード後も途中から再開できること
5. 斜線が引かれたセットを DB で UPDATE する際、同一セッション内の複数セットを一括 UPDATE すること（パフォーマンス考慮）
6. `middleware.ts` で未認証ユーザーを `/` へリダイレクトし、認証済みユーザーが `/` にアクセスした場合は `/mypage` へリダイレクトすること
